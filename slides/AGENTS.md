# Deck context

## Audience

Senior partner engineers and model or product engineers evaluating a working document-automation prototype. They need the operating decision, technical reasoning, measured evidence, and useful product feedback without a long setup.

## Tone

Calm, technical, candid. Explain why each design choice exists and how it affects partner operations. Prefer concrete measurements and reproduced errors to broad claims.

## Visual direction

Use a light operational layout that matches the application. Keep type large enough for a shared screen. Use dark green for accepted paths, amber for review or caveats, red only for reproduced failures, and blue for system boundaries. Avoid gradients, decorative shapes, nested cards, and animation.
